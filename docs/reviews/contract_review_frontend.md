# 前端合约评审报告

## 评审结论：需修改

## 发现问题

1. **交互闭环问题**：注册流程中，当钱包创建失败时，PRD 6.1节描述返回“钱包初始化中，请稍后查看”，但OpenAPI中`POST /users`接口的500错误响应未明确区分“用户创建成功但钱包失败”这一特定场景。前端无法根据现有错误响应（一个通用的500错误）来向用户展示“钱包初始化中”的友好提示，也无法引导用户后续如何操作（如刷新页面、重新登录等）。

2. **类型一致性问题**：
   - **问题1**：`LoginResponse`中的`user`字段类型为`UserProfile`，而`UserProfile`中的`wallet_address`字段被定义为`nullable: true`。这与PRD中FR-03“登录成功，返回...钱包地址”的描述存在潜在矛盾。如果钱包创建失败或状态为pending，登录时`wallet_address`可能为null，前端需要明确如何处理（例如显示“地址生成中”）。
   - **问题2**：`UserResponse`（用于`GET /me`和`POST /users`的响应）继承了`UserProfile`，因此其`wallet_address`同样可为null。但PRD FR-02描述注册成功应返回钱包地址。如果钱包创建是异步或可能失败，注册成功响应中的钱包地址也可能为null，这与当前描述不符。

3. **移动端接口需求**：所有接口均未考虑移动端常见的需求，如：
   - **分页与长列表**：当前无列表接口，暂无影响。
   - **图片上传**：当前无相关需求，暂无影响。
   - **Token刷新机制**：OpenAPI定义了Bearer Token认证，但未提供Token刷新接口（`/auth/refresh`）。移动端应用需要长期维持登录态，仅靠一个有时效的Access Token无法满足需求，需补充刷新机制或明确Access Token为长期有效（不推荐）。

4. **前端可实现性细节**：
   - **密码复杂度提示**：PRD FR-02要求密码“至少8位，包含字母和数字”，但OpenAPI中`RegisterUserRequest`的`password`字段仅通过`minLength: 8`和`format: password`描述，未在schema中通过`pattern`等方式明确表达“包含字母和数字”的规则。这可能导致前端校验规则与后端不一致，引发不必要的错误提交。
   - **验证码重发倒计时**：PRD FR-01约束“同一邮箱60秒内只能请求一次”，OpenAPI通过429状态码和幂等性描述来体现。前端需要知道具体的冷却时间（60秒）来实现重发按钮的倒计时功能，此信息在OpenAPI接口描述中未明确，不利于前端开发。
