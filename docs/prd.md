# H Wallet MVP 产品需求文档

## 1. 文档概述
本文档定义了 H Wallet MVP 版本的核心需求，聚焦于邮箱注册、自动创建 OKX Agent Wallet 和登录流程。

## 2. 产品愿景
为 Web3 用户提供便捷的邮箱注册入口，自动创建 OKX Agent Wallet，实现无缝的登录体验，并为后续 Slack 多 Agent 工作流协作奠定基础。

## 3. 用户故事
作为新用户，我希望通过邮箱注册 H Wallet，以便系统能自动为我创建一个 OKX Agent Wallet，后续可以通过邮箱和密码登录使用。

## 4. 功能需求

### P0 优先级（MVP 核心）

#### FR-01：发送邮箱验证码
- **描述**：用户输入邮箱后，系统向该邮箱发送一个6位数字验证码，用于验证邮箱所有权。
- **输入**：邮箱地址（符合邮箱格式规范）。
- **处理**：
  1. 校验邮箱格式。
  2. 生成6位随机数字验证码。
  3. 将验证码与邮箱、发送时间关联存储，并设置5分钟有效期。
  4. 调用邮件服务发送验证码（MVP 阶段可模拟发送，在日志中打印）。
- **输出**：发送成功或失败提示。
- **约束**：
  - 同一邮箱60秒内只能请求一次。
  - 验证码有效期5分钟。
  - 需具备幂等性，短时间重复请求返回相同结果。

#### FR-02：校验邮箱验证码并注册
- **描述**：用户输入邮箱、收到的验证码和密码，完成注册。
- **输入**：邮箱、验证码、密码（密码需满足复杂度要求：至少8位，包含字母和数字）。
- **处理**：
  1. 校验邮箱格式、验证码格式、密码复杂度。
  2. 校验验证码是否正确且在有效期内。
  3. 检查邮箱是否已被注册。
  4. 密码加盐哈希存储。
  5. 调用 `POST /wallets` 接口，触发系统自动创建 OKX Agent Wallet。
  6. 将用户信息（邮箱、密码哈希、钱包地址）持久化存储。
- **输出**：注册成功，返回用户唯一标识（如 user_id）及自动创建的钱包地址。
- **约束**：
  - 邮箱唯一。
  - 验证码一经使用立即失效。

#### FR-03：邮箱密码登录
- **描述**：已注册用户使用邮箱和密码登录。
- **输入**：邮箱、密码。
- **处理**：
  1. 校验邮箱格式。
  2. 根据邮箱查找用户记录。
  3. 校验密码哈希是否匹配。
  4. 生成登录态（如 JWT Token），包含用户ID等信息。
- **输出**：登录成功，返回 Token 及用户基本信息（如 user_id, email, wallet_address）。

#### FR-04：自动创建 OKX Agent Wallet
- **描述**：在用户成功注册后，系统自动调用 OKX 相关服务（或模拟服务）为该用户创建一个 Agent Wallet。
- **触发**：`FR-02` 注册流程成功后同步调用。
- **处理**：
  1. 接收来自注册流程的请求（包含新用户的 user_id）。
  2. 调用外部服务或内部模拟服务生成钱包地址和关联信息。
  3. 将生成的钱包地址与用户绑定并持久化。
- **输出**：钱包地址。
- **约束**：
  - 必须确保与用户记录的原子性（用户创建成功，则必须尝试创建钱包；若钱包创建失败，应有明确失败状态和重试/补偿机制）。
  - 此过程对用户透明。

### P1 优先级（MVP 增强）

#### FR-05：获取登录用户信息
- **描述**：已登录用户可获取自己的基本信息。
- **输入**：有效的登录 Token（通过 Authorization Header 传递）。
- **处理**：验证 Token 有效性，返回对应用户信息。
- **输出**：用户信息（user_id, email, wallet_address, 注册时间等）。

#### FR-06：统一的错误处理与响应格式
- **描述**：所有 API 接口遵循统一的错误响应格式，包含明确的状态码和错误信息。
- **约束**：定义清晰的 HTTP 状态码和业务错误码映射关系。

## 5. 非功能性需求

### NF-01：API 幂等性
- 对于创建性操作（如注册、发送验证码），在请求参数相同时，多次调用应产生相同结果，避免重复创建。

### NF-02：错误码规范
- 系统需定义清晰的业务错误码，便于前端和后续 Agent 识别处理。
- HTTP 状态码与业务错误码分离。业务错误详情在响应体中返回。

### NF-03：日志与可追踪性
- 关键业务节点（如发送验证码、注册、登录、创建钱包）必须打印结构化日志。
- 日志需包含唯一请求 ID (`request_id`)，实现全链路追踪。
- 日志级别规范：INFO 记录正常操作，ERROR 记录失败和异常。

### NF-04：安全性
- 用户密码必须加盐哈希存储（如使用 bcrypt）。
- 验证码等敏感信息不得在日志中明文打印。
- API 接口需有适当的限流策略（MVP 可先实现简单的频率限制）。

### NF-05：性能
- 核心接口（登录、注册）P99 响应时间应小于 1 秒。
- 验证码发送接口因依赖外部服务，可适当放宽，但应有超时控制（如 3 秒超时）。

## 6. 流程与状态

### 6.1 注册流程
1.  **步骤1（发送验证码）**：`FR-01`
    - 用户输入邮箱 -> 请求 `POST /verification-codes` -> 系统发送验证码 -> 返回成功。
    - **失败兜底**：前端提示“发送失败，请稍后重试”。系统记录错误日志。
2.  **步骤2（提交注册）**：`FR-02`
    - 用户输入邮箱、验证码、密码 -> 请求 `POST /users` -> 系统校验并创建用户 -> **同步调用** `FR-04` 创建钱包 -> 返回用户及钱包信息。
    - **失败兜底**：
        - 验证码错误/过期：返回具体错误，引导用户重新获取。
        - 邮箱已注册：提示直接登录。
        - **钱包创建失败**：此为关键风险点。系统应将用户标记为“钱包创建失败”状态，并返回注册成功但提示“钱包初始化中，请稍后查看”。后台应有异步重试机制。

### 6.2 登录流程
1.  **步骤**：`FR-03`
    - 用户输入邮箱、密码 -> 请求 `POST /auth/login` -> 系统校验 -> 生成 Token -> 返回 Token 及用户信息。
    - **失败兜底**：邮箱或密码错误返回统一提示“邮箱或密码错误”，避免枚举用户。账户锁定等高级功能 MVP 暂不涉及。

## 7. 假设、风险与待确认项

### 假设
1.  MVP 阶段用户量不大，暂不考虑分库分表。
2.  “自动创建 OKX Agent Wallet” 在 MVP 阶段使用模拟服务实现，返回一个模拟的区块链地址。
3.  邮件发送服务在 MVP 阶段通过日志模拟，不集成真实邮件服务。

### 风险与协商建议
1.  **风险**：`FR-04` 自动创建钱包是核心依赖。若真实对接 OKX，其 API 稳定性、速率限制、网络延迟可能影响注册成功率和用户体验。
    - **建议**：MVP 阶段强烈建议使用模拟服务，返回固定或随机生成的地址。将真实集成作为 P1 或 P2 后续迭代。
2.  **风险**：注册流程中，用户创建成功但钱包创建失败，导致用户状态不一致。
    - **建议**：按 `6.1` 的兜底方案处理，设计“钱包状态”字段（`active`, `pending`, `failed`），并实现一个简单的后台任务重试失败的创建请求。
3.  **待确认**：后续 Slack Agent 工作流所需的用户权限、可访问的 API 范围。
    - **建议**：在用户表和 Token 中预留 `scope` 或 `role` 字段，为后续扩展做准备。

## 8. 后续迭代方向（非 MVP）
1.  忘记密码/重置密码流程。
2.  真实的邮件服务集成。
3.  真实的 OKX Agent Wallet 服务集成。
4.  Slack Agent 授权与 API 访问控制。
5.  用户 Profile 管理。
