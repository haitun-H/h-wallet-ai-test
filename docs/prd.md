# H Wallet MVP 产品需求文档

## 1. 文档概述
本文档定义了 H Wallet MVP 版本的核心需求，聚焦于邮箱注册、自动创建钱包和登录流程。本版本为最小可行产品，旨在验证核心业务流程。

## 2. 产品愿景
为 Web3 用户提供简单、安全的入口钱包，通过邮箱即可快速创建和管理 OKX Agent Wallet。

## 3. 目标用户
- Web3 新手用户
- 寻求便捷钱包创建体验的用户
- 需要邮箱作为主要身份标识的用户

## 4. 功能需求

### P0 - MVP 核心流程

#### FR-01: 发送邮箱验证码
**优先级**: P0
**描述**: 用户输入邮箱后，系统发送6位数字验证码到该邮箱。
**输入**: 邮箱地址
**处理**:
1. 校验邮箱格式
2. 检查该邮箱60秒内是否已发送过验证码（防刷）
3. 生成6位数字验证码，存储并设置5分钟有效期
4. 发送邮件（MVP阶段可模拟发送，日志记录）
**输出**: 发送成功或失败提示
**约束**:
- 同一邮箱60秒内只能请求一次验证码
- 验证码有效期为5分钟
- 验证码为6位纯数字

#### FR-02: 用户注册
**优先级**: P0
**描述**: 用户使用邮箱、验证码、密码完成注册，系统自动创建钱包。
**输入**: 邮箱、验证码、密码、确认密码
**处理**:
1. 校验验证码正确性和有效性
2. 校验邮箱是否已注册
3. 校验密码复杂度（至少8位，包含字母和数字）
4. 校验密码和确认密码一致
5. 创建用户记录，密码加密存储
6. **异步触发**钱包创建任务
7. 注册成功，返回用户基本信息（钱包地址可能为null）
**输出**: 注册成功响应，包含用户ID、邮箱、钱包状态
**约束**:
- 密码需加密存储（bcrypt）
- 钱包创建为异步流程，不影响注册主流程
- 注册成功后自动登录（返回access_token）

#### FR-03: 用户登录
**优先级**: P0
**描述**: 已注册用户使用邮箱和密码登录。
**输入**: 邮箱、密码
**处理**:
1. 校验邮箱格式
2. 验证邮箱和密码匹配
3. 生成JWT token（有效期7天）
4. 返回用户信息和token
**输出**: 登录成功响应，包含用户信息和access_token
**约束**:
- 支持长期登录态（token有效期7天）
- 登录失败需明确提示（邮箱未注册/密码错误）

#### FR-04: 自动创建 OKX Agent Wallet
**优先级**: P0
**描述**: 用户注册成功后，系统自动为其创建 OKX Agent Wallet。
**处理**:
1. 监听用户注册成功事件
2. 调用钱包创建服务（MVP阶段模拟）
3. 生成钱包地址和私钥（模拟）
4. 安全存储钱包信息（模拟）
5. 更新用户钱包状态和地址
**输出**: 钱包创建成功或失败
**约束**:
- MVP阶段完全模拟实现
- 钱包创建为异步任务
- 需处理创建失败的重试机制（最多3次）

#### FR-05: 获取当前用户信息
**优先级**: P0
**描述**: 已登录用户获取自己的完整信息。
**输入**: Bearer Token
**处理**:
1. 验证token有效性
2. 获取用户ID
3. 查询用户完整信息
4. 返回用户资料（包含钱包状态）
**输出**: 用户完整信息
**约束**: 需要有效的登录token

### P1 - 增强功能

#### FR-06: Token刷新机制
**优先级**: P1
**描述**: 在access_token过期前刷新，获取新的token。
**输入**: refresh_token
**处理**:
1. 验证refresh_token有效性
2. 生成新的access_token（有效期7天）
3. 返回新的token对
**输出**: 新的access_token和refresh_token
**约束**:
- refresh_token有效期为30天
- 每次刷新后生成新的refresh_token

#### FR-07: 查询钱包创建状态
**优先级**: P1
**描述**: 用户主动查询钱包创建进度。
**输入**: Bearer Token
**处理**:
1. 验证token有效性
2. 查询用户钱包状态
3. 返回钱包状态（pending/active/failed）和详细信息
**输出**: 钱包状态信息
**约束**: 需要用户已登录

## 5. 非功能性需求

### NF-01: 性能要求
- 注册接口平均响应时间 < 500ms（不含异步钱包创建）
- 登录接口平均响应时间 < 300ms
- 验证码发送接口平均响应时间 < 200ms

### NF-02: 安全性要求
- 所有敏感接口必须使用HTTPS
- 密码必须加密存储（bcrypt）
- JWT token使用HS256算法，密钥长度至少32位
- 验证码防刷机制（60秒冷却时间）
- 输入参数必须验证和过滤

### NF-03: 错误处理
- 统一的错误响应格式
- 业务错误码体系（见错误码表）
- 详细的错误日志记录（包含request_id）
- 用户友好的错误提示

### NF-04: 幂等性要求
- 注册接口支持幂等（基于邮箱）
- 验证码发送接口支持幂等（60秒内同一邮箱返回相同验证码）
- 钱包创建任务支持幂等（基于用户ID）

### NF-05: 可追踪性
- 所有请求必须生成唯一的request_id
- 关键业务操作记录审计日志
- 分布式链路追踪（预留接口）
